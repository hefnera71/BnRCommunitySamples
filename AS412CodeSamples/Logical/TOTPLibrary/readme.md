# Library TOTPlib

## About

This libary implements a "Time-based One-Time Password" (TOTP) algorithm as defined in RFC 6238.

**WARNING: when using ArSim, code validation may fail!**

The reason behind: the ArSim RTC is a "simulated one" that is not synchronized to the real absolute time / the Windows clock.
So the timestamp generated in the simulated PLC could have a (to) high deviation to the real time, which will lead to non synchronized codes.
This is not a bug! For reliable testing, please use a hardware-based PLC.

## How it works
### the algorithm
TOTP uses HMAC-SHA1 and the Unix epoch timestamp (seconds since 01.01.1970) as counter as defined in RFC 6238.
This impementation works with a 6-digit code, which is most common used with Authenticator apps.
As the code generation is time-based like mentioned, the PLC realtime clock (RTC) has to be set correct for proper functionality - I used the right timezone settings + NTP client functionality of the PLC to ensure that the base clock value of the external generator and the PLC is the same.

### shared secret
The external generator (Authenticator App) needs to have the base32-coded HMAC-SHA1 shared secret.
Such secret is generated by the function block automatically, based on two input strings of maximum size 32 byte.
One string can be used in any way wanted, the second one can be use either free like the first string, or as the name of a own certificate which is part of the AS project.
If using this 2nd string as certificate name, the function block internally reads out some information of this certificate, and uses that information as 2nd input for the secret generation.
The secret itself is generated as ```Base32(HMAC_SHA1(<input1 + SALT>,<input2 + SALT>))```

By having these 2 strings for shared secret generation, you have various possibilities to
* provide always the same code
* provide different users different codes (by using different string per user)
* provide different codes on different hardware (e.g. by using for example the MAC address or the serialnumber as one of the input strings)
* provide different codes by using different certificates
* disable the usage of outdated or compromised accounts (e.g. by removing / changing the certificate used for shared secret generation)
* ....

### code validity duration
Common external generators (like Google Authenticator) used for 2-factor authentication are producing codes with a fixed validitiy time of 30 seconds.
Therefore, the function block also works with a validity time of 30 seconds.
To avoid clock synchronisation jitter based problems, you can configure if just this single code is accepted, or additionally the previous + next code, or additionally the two previous + two next codes. 

In PLC environment maybe you have different needs, e.g. like a "password of the day" - that's possible by configuring the number of seconds how long the token should be the same.
Then you need to use a external generator which is also able to generate tokens with longer validation time >> but please note, that the well-known Authenticator apps don't support that, they just work with 30 second code!

But the're other implementations out there, so no doubt if you want to use the functionality not as a 2-factor authentication mechanism with a "just 30 second valid" code.
For my tests, I used the following online token generator: https://totp.danhersam.com/

## function block interface
He're the function block VerifyTOTPCode inputs and outputs with some short usage comments.
```
	VAR_INPUT
		enable : BOOL; (*enable the function block*)
		code : UDINT; (*the external code from authenticator app*)
		tolerance : USINT; (*0 = accepts just the actual 30 seconds token, 1 = accepts actual token and previous + next one, 2 = accepts actual token plus last 2 and next 2 ones*)
		validity : UINT; (*0 - 30 = default = 30 seconds; > 30 code validity in seconds. WARNING! Using values > 30 seconds does not work with common Authenticator apps!*)
		pId1 : UDINT; (*pointer to own free string[32] value which is used for shared secret generation*)
		pId2 : UDINT; (*pointer to own free string[32] value or name of certificate name which is used for shared secret generation*)
		id2IsOwnCertificate : BOOL; (*1 = use the id2 as certificate name, 0 = use id2 as free value*)
		otpauthLabel : STRING[64]; (*label of the otpauth link, oten used with following pattern <IssuerName:AccountName>*)
	END_VAR
	VAR_OUTPUT
		codeValid : BOOL; (*true = code is valid, false = code is invalid*)
		phase : USINT; (*0 = starting, 1 = reading cert, 2 = preparing secret, 3 = READY TO CHECK CODES*)
		status : UINT; (*FB status - 65535 if running, 65534 if disable, error if anything else*)
		statusID : DINT; (*Extended status (internal FB call StatusID in case of FB call error)*)
		pCertValidFrom : UDINT; (*pointer to DTStructure -> certificate from date (just for information/external reaction! FB internally this is ignored!)*)
		pCertValidTo : UDINT; (*pointer to DTStructure -> certificate to date (just for information/external reaction! FB internally this is ignored!)*)
		pOtpSharedB32S : UDINT; (*pointer to string[128] holding the otp shared secret - string has to have size 128!!*)
		pOtpauthLink : UDINT; (*pointer to string[256] holding the otpauth link - string has to have size 255!!*)
	END_VAR
```

### additional function block interface information
Why have I used addresses ("pointers") as inputs / outputs instead of suitable data types?
The reason is simple: just to avoid to expose sensitive data directly via AS Watch by just putting the function block instance into a watch window.
Using addresses is a bit more inconvenient and needs maybe a line of code more, but it decouples the variable which is holding the data from the functionality using / producing this data.
Sure, this is absolutely no "high-end security feature"! But it's just another little building block to reduce the risk of exposing data by mistake.

### What is the pOtpauthLink and pOtpSharedB32S output, what is the otpauthLabel input?
To connect to a Authenticator App, you have to exchange once the shared secret and some account information.
Most often that's done by a QR code, and such a QR code contains a otpauth link.
The common format of a TOTP otpauth-link looks like: otpauth://totp/<otpauthLabel servicename:accoutname>?<parameter secret>
And such a link is provided by the function block output .pOtpauthLink, so that you can share this link with an Authenticator App.
If using some online token generator but not an app on a mobile or similar, you maybe need just the shared secret: that's provided by the function block output .pOtpSharedB32S
As described above, for some reason these outputs are just pointers. To read the data, you have to access that pointer for example by reference variable of suitable type.
 
### Usage example
The task "TstToptLib" contains a simple example how to call the function block.
Additionally, it includes a function block call out of a 2nd independent libary: WsQRLib.

This library contains a function block which is providing a webservice (based on AsHttps) that generates a QR code out of a string.
Please see the readme of the library WsQRlib for details (also included in that repository).



```
PROGRAM _INIT

	id1 := 'MyVerySecretString';	 
	id2 := 'MyCert.cer';

END_PROGRAM

PROGRAM _CYCLIC
	// sample of TOTP code generator call
	
	VerifyTOTPCode_0.enable := enable;						// enable the FB
	VerifyTOTPCode_0.code := code;							// input your code generated by your app to check it
	VerifyTOTPCode_0.pId1 := ADR(id1);						// 1st string used for shared secret
	VerifyTOTPCode_0.pId2 := ADR(id2);						// 2nd string used for shared secret
	VerifyTOTPCode_0.id2IsOwnCertificate := TRUE;			// 2nd string is used as certificate name, so for secret generation the certificate serial number is used
	VerifyTOTPCode_0.otpauthLabel := 'BRCommunity:Alex';	// the label for the otpauth link generated by this FB
	VerifyTOTPCode_0.tolerance := 1;						// allow actual code + previous + next one
	VerifyTOTPCode_0.validity := 0;							// use default = 30 seconds
	VerifyTOTPCode_0();										// call the FB

	// phase 3 reached - everything should be prepared and function should run under normal condition
	IF VerifyTOTPCode_0.phase = 3 THEN

		// now you can prove the codes
		IF VerifyTOTPCode_0.codeValid = TRUE THEN
			// do what ever you want to do if a valid code was entered
		ELSE
			// do what ever you want to do if a valid code was entered
		END_IF

		// extended sample - if FB is running under normal condition, prepare the possible usage of QR webservice for linking authenticator app
		enaQR := TRUE;
		// just set additionally "useQR" to true in watch window to enable QR code generation
	ELSE
		// disable webservice if not in phase 3 (this sample would not workthen)
		enaQR := FALSE;
	END_IF
	
	// sample of QRCode webservice call together with TOTP code
	// please note, that this FB is an own libary that can also be used independent of TOTP / otpauth
	QRCodeWebservice_0.enable := enaQR AND useQR;						// enable the FB
	QRCodeWebservice_0.pDataString := VerifyTOTPCode_0.pOtpauthLink;	// use the otpauth link generated by TOTP FB as data for the QR code
	QRCodeWebservice_0.webserviceName := 'TOTPQR.cgi'; 					// name how the webservice can be reached, in this sample -> https://<IPAdress of PLC>/TOTPQR.cgi
	QRCodeWebservice_0();												// call the FB

	
	// access the output pointers to get access to certificate date information (if certificate is used), and to see the shared secret in base32 encoding + the otpauth link
	// !!! WARNING !!!
	// !!! THIS CODE IS JUST MEANT TO CLARIFY HOW TO ACCESS THE HIDDEN FB OUTPUT DATA
	// !!! DONT DO THIS IN YOUR PRODUCTIVE CODE (IF NOT ABSOLUTELY NECCESSARY) TO AVOID EXPOSING YOUR SECRET DATA VIA WATCH
	cValidFrom ACCESS VerifyTOTPCode_0.pCertValidFrom;
	cValidTo ACCESS VerifyTOTPCode_0.pCertValidTo;
	b32Secret ACCESS VerifyTOTPCode_0.pOtpSharedB32S;
	otpAuthLink ACCESS VerifyTOTPCode_0.pOtpauthLink;
	 
END_PROGRAM

PROGRAM _EXIT

	VerifyTOTPCode_0(enable := FALSE);
	QRCodeWebservice_0(enable := FALSE);
	 
END_PROGRAM
```
